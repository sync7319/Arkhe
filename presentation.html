How Arkhe Works

Near-deterministic results —
how and why

AST first, LLM second

tree-sitter extracts every function, class, import, export, and call edge before any LLM call. The analyst prompt includes the exact AST-extracted names and is instructed: "Only use names that appear verbatim. Never invent." The LLM annotates ground-truth structure — it cannot hallucinate what tree-sitter already found.

Deterministic pre-passes replace LLM guesswork

Dead code: call graph → regex cross-file → Vulture, zero LLM. Security pass 1: Bandit (CWE-tagged) + Semgrep (30+ languages) run before the LLM. Test gap phase 1: static function→test cross-reference. The LLM only handles what static analysis genuinely cannot: semantic logic and narrative.

Quality scoring + auto-rescore

Every per-file analysis scored 0–1: AST name coverage (50%) + length vs file size (30%) + structural keywords (20%). Files scoring below 0.35 are automatically re-sent to the LLM with +256 extra output tokens. Prevents garbage-in-garbage-out before synthesis.

SHA-256 content cache — 1 changed file = 1 LLM call

Every file is cached by SHA-256 of its content. Re-run Arkhe on a 200-file repo after touching 1 file: 199 files served from SQLite instantly, 1 file hits the LLM. Cooldowns also persist to SQLite — a rate-limited model stays cooling across process restarts.

Result: consistent, hallucination-resistant output even when cheap models handle 80% of the work.

Feature

Architecture Map —
CODEBASE_MAP.md

What it is

A narrative architecture document generated automatically for any repo. Explains the codebase to a developer who has never seen it — covering architecture, data flows, module responsibilities, key abstractions, and hidden gotchas.

Sections generated
  • →System architecture overview
  • →Module-by-module breakdown
  • →Data flow diagrams (narrative)
  • →Key classes and functions
  • →External dependencies and their roles
  • →Gotchas, quirks, and non-obvious invariants

How it works

1. Scan

Walk repo, tokenize every file

2. Parse

tree-sitter: extract all functions, classes, imports, call graph

3. Analyze

Cheap models describe each file in parallel batches

4. Synthesize

Heavy pool (NVIDIA Nemotron-253B first) reads all batch reports and AST-grounded imports to write the final map — one call, never per-file

5. Write

CODEBASE_MAP.md saved to docs/ and S3

Why it matters: onboarding a new dev to a large repo takes hours — this doc takes seconds to generate and never goes stale.

Feature

Interactive Dependency Map —
DEPENDENCY_MAP.html

What it is

A force-directed D3.js graph rendered as a self-contained HTML file. Every file in the repo is a node. Every import is a directed edge. Fully interactive — click any node to see what it imports and what imports it.

What you can see
  • →Complexity heatmap (radon CC for Python, heuristic for others)
  • →Hub files — highest in-degree, most depended-upon
  • →Circular dependencies — instantly visible as cycles
  • →Isolated and leaf nodes — files with no connections
  • →Clickable info panel: defines, imports, imported-by
  • →Mutual dependency paths drawn separately (no overlap)

How it works (zero LLM cost)

Parse

tree-sitter AST extracts every import statement from source

Resolve

Dotted module → file path (Python); relative path strings for JS/TS via _resolve_import_js

Score

radon cyclomatic complexity for Python; token+import+function heuristic for all other languages

Render

Nodes/links JSON injected into D3.js template; self-contained HTML, no server needed to view

Entirely deterministic — no LLM involved at any step.

Feature

OWASP Security Report —
SECURITY_REPORT.md

What it covers

  • →Hardcoded credentials (literal secrets in source)
  • →SQL / command injection (f-strings in queries, shell=True)
  • →Path traversal (user-supplied paths to open/os.path)
  • →Unvalidated inputs reaching dangerous sinks
  • →Missing auth on sensitive HTTP routes
  • →Insecure deserialization (pickle.loads, yaml.load)
  • →Sensitive data in logs or error messages
  • →Weak crypto (MD5/SHA1 for passwords, hardcoded IV)
  • →SSRF (user-controlled URLs to requests without allow-list)

Three-pass approach

Pass 1 — Bandit

Python static analysis. CWE-tagged findings. Full-file coverage (no truncation). Zero LLM cost. Skips B404/B603/B607 (subprocess noise) — only real issues.

Pass 2 — Semgrep

Multi-language AST-based scan (30+ languages). OWASP Top 10 rules. Runs on full repo root. Zero LLM cost. Gracefully skips if binary not installed.

Pass 3 — LLM semantic

Catches logic flaws, auth issues, and SSRF that static tools miss. Context-aware prompt prevents false positives on tool design. Uses traversal-role model — no expensive synthesis model needed.

Output: FILE / SEVERITY / ISSUE / CODE / FIX for every finding — structured for automated parsing.

Feature

Dead Code & Blast Radius —
precision static analysis

Dead Code — DEAD_CODE_REPORT.md

Functions and classes defined but never called outside their own file. Across all 7 languages. Skips dunders, framework-decorated symbols (@app.route, @router.get, @pytest.fixture), and exported symbols in __all__.

Pass 1 — Call graph

tree-sitter extracts function→callee relationships. Any function appearing as a callee is immediately marked live. Zero false positives across module boundaries.

Pass 2 — Regex cross-file

For every defined symbol, scan all other files for name occurrences. Decorator-aware: checks 3 preceding lines for @app. @router. @pytest. etc.

Pass 3 — Vulture (Python)

AST-based analysis at 80% confidence. Used to remove false positives from pass 2 — conservative filter, not a primary detector.

194 symbols scanned → 4 genuine findings (28 before false-positive reduction)

Blast Radius — PR_IMPACT.md

Given a PR, Arkhe diffs HEAD against a base branch, then walks the dependency graph in reverse using NetworkX to find every file that directly or transitively imports the changed files.

git diff

Get list of changed files: HEAD vs base branch

Reverse graph

Build NetworkX DiGraph; walk ancestors() to get full transitive dependents — not just direct importers

LLM summary

Heavy pool writes plain-English impact: What changed / Downstream impact / Risk level (LOW / MEDIUM / HIGH)

Without this: devs merge a shared utility change not knowing it breaks 12 downstream modules.
