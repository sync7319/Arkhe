SLIDE 3 — USING POWERFUL AI ON CODEBASES IS EXPENSIVE
Presenter: OM
================================================================================

  "The obvious approach is: just send every file to GPT-4o. Problem solved.
   Except — look at these numbers."

  "A 500-file repo generates roughly two million input tokens. At GPT-4o's
   rate of $15 per million tokens, that's $30 per run. Per run. Every time
   you want updated docs."

  "And it's not just cost — you hit rate limits before the analysis even
   finishes. The model times out. You get partial output."

  "Here's the key insight we built around: expensive models are overkill
   for 80% of this work. Analyzing a 10-line config file doesn't need
   GPT-4o. Arkhe routes each task to the cheapest capable model — and uses
   free API tiers by default. Most users pay nothing."

SLIDE 6 — PIPELINE: RIGHT MODEL FOR EVERY JOB
Presenter: SHREEYUT
================================================================================

  "This is the engineering core of Arkhe. The pipeline has six stages,
   and the critical design decision is that we split them into two
   categories: deterministic work and LLM work."

  "Everything in green — scan, parse, the dependency graph, dead code
   detection, security static passes — runs at zero LLM cost. It's
   just code, running fast and producing ground-truth results."

  "The LLM only handles what static analysis genuinely cannot: writing
   the narrative, understanding semantic logic in security findings,
   and generating plain-English summaries."

  "This isn't just a cost optimization. It makes the output more
   accurate. The LLM is given AST-extracted ground truth — actual
   function names, actual import paths — and told to annotate it.
   It can't hallucinate structure that tree-sitter already found."

  "The result is a pipeline that's mostly deterministic, with LLM
   intelligence applied only where it adds unique value."

SLIDE 7 — TIERED MODEL ROUTING
Presenter: SHREEYUT
================================================================================

  "This is the model routing system that makes the cost story work."

  "We have five tiers plus a heavy pool, and every file gets routed
   based on its token count. A 200-line config file goes to Tier 1 —
   a tiny cheap model, basically free. A 2,000-line core module goes
   to Tier 3 or 4 where we need more capability."

  "NVIDIA Nemotron-253B is our primary model on the web platform — it's
   on a free dev tier, which means our server cost for LLM inference
   is currently zero."

  "The auto-cascade is what makes this robust: if the primary model
   hits a rate limit, Arkhe immediately drops to the next model in
   that tier without any waiting, any manual config, or any error
   surfaced to the user. Cooldowns persist in SQLite across restarts."

  "This is why a 500-file repo doesn't cost $30 per run. Most files
   hit cheap or free tiers. The expensive models only see the files
   that actually need them."

SLIDE 8 — NEAR-DETERMINISTIC RESULTS
Presenter: SHREEYUT
================================================================================

  "This slide is about output quality — specifically, how we prevent
   the LLM from hallucinating things."

  "The AST-first approach is the key one. Before any LLM call, tree-sitter
   has already extracted every function name, class, import, and call
   relationship in the file. We inject those verbatim into the prompt
   and tell the model: use only these names. Never invent."

  "Early versions of Arkhe would produce docs that mentioned functions
   called 'init_client' or 'TreeSitterParser' — functions that didn't
   exist. Classic LLM hallucination. The AST grounding eliminated that."

  "The quality scoring system is something I'm particularly proud of.
   Every file analysis gets scored zero to one based on how much it
   actually references the real code structure. If a file scores below
   0.35, it automatically gets re-analyzed with a larger output budget.
   This catches the cases where the model produced something too vague
   or too short to be useful."

  "And the SHA-256 cache means that on a re-run, we only spend compute
   on what actually changed. Touch one file, one LLM call, not two hundred."

SLIDE 10 — DEPENDENCY MAP DEEP DIVE
Presenter: SHREEYUT
================================================================================

  "The dependency map is the only output that's completely deterministic —
   no LLM involved at any step. Same repo, same commit, always produces
   the same graph."

  "It's an interactive D3.js force graph. Every file is a node. Every
   import statement is a directed edge. The color of each node is its
   cyclomatic complexity — green for simple, red for complex."

  "The import resolution was harder than it sounds. Python has dotted
   module paths and relative imports. JavaScript has relative paths
   with and without extensions, index.js conventions, and TypeScript
   path aliases. We handle all of these with separate resolution logic
   per language."

  "What makes this practically useful: you can instantly spot the hub
   files — the ones everything else imports. Those are your highest-risk
   files for changes. You can see circular dependencies as cycles in the
   graph. And you can click any node and see exactly what its blast
   radius would be if you changed it."

SLIDE 11 — SECURITY REPORT DEEP DIVE
Presenter: SHREEYUT
================================================================================

  "The security report uses three passes — and the order matters."

  "Bandit runs first on Python files. It's a static AST-based tool that
   produces CWE-tagged findings. We tuned it to skip the noisy rules —
   B404, B603, B607 — which flag subprocess usage. Arkhe is an analysis
   tool that runs subprocesses by design. We'd be flagging our own code."

  "Semgrep runs second. It covers 30+ languages with OWASP Top 10
   patterns. This is where we catch things Bandit can't — SQL injection
   in TypeScript, path traversal in Go, SSRF in any language."

  "The LLM runs third, and only for what static analysis genuinely misses:
   authentication logic, SSRF where the URL construction is spread across
   multiple functions, business logic that's technically valid code but
   semantically wrong."

  "The output format — FILE / SEVERITY / ISSUE / CODE / FIX — is
   structured so it can be parsed programmatically. You could pipe this
   into a CI step that fails the build on HIGH severity findings."


================================================================================
SLIDE 12 — DEAD CODE + BLAST RADIUS
Presenter: SHREEYUT
================================================================================

  "Two features on one slide because they share the same underlying
   data structure: the call graph."

  "Dead code detection: we scan every symbol in the codebase and check
   whether it appears as a callee in the call graph. If yes, it's live.
   If no, we do a cross-file regex scan. If still nothing, Vulture gets
   a look. Three passes."

  "The number I'm proud of: we went from 28 flagged symbols down to 4
   genuine dead code findings. That 24-symbol reduction came entirely
   from the decorator awareness — FastAPI routes decorated with @app.get
   look dead because nothing in Python code calls them directly. They're
   called by the HTTP framework. We detect those decorators and skip them."

  "Blast radius is the reverse problem: instead of 'what calls this?'
   it's 'what does this break if it changes?' We use NetworkX to walk
   the dependency graph in reverse — not just direct importers, but
   full transitive dependents. Change a shared utility and you'll see
   every module in the chain, all the way out."

  "Without this, you're merging PRs blind. You don't know you just broke
   12 downstream modules until CI fails — or worse, until production."

GENERAL TIPS
================================================================================

- Don't read the slides. The slides are for the audience. Your job is
  to add context, decisions, and story that isn't on screen.

- When a slide has numbers, say them out loud with emphasis. Pause
  after "zero tools" or "$12 a year" — let it land.

- If asked "why not just use ChatGPT?": token limits, no full repo
  context, no persistent output, no static analysis layer. ChatGPT
  sees a slice. Arkhe processes the whole thing.

- If asked about competitors: CodeSee is the cleanest answer. Real
  company, real funding, real shutdown. Their users need something.

- If asked about monetization: BYOK is the key. We never pay inference
  costs for CLI users. The margin on Pro is essentially the full $15.

- If asked "what's next?": GitHub webhook integration so it runs
  automatically on every PR. That's the retention mechanism.
