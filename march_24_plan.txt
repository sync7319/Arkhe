1. networkx — Real Graph Analysis for Dependencies & Impact
   Currently: Hand-rolled adjacency lists in agents/impact_agent.py:_build_reverse_map().
   Impact analysis is 1-level only. Graph stats in server/app.py manually compute
   hubs, cycles, isolated nodes.

   Recommendation: Build a nx.DiGraph from the existing graph["nodes"] and graph["links"]
   data. This unlocks:
   - Transitive impact: nx.ancestors(G, file) — full blast radius, not just direct importers
   - PageRank: nx.pagerank(G) — rank files by structural importance for context picker
   - Betweenness centrality: nx.betweenness_centrality(G) — find critical junction files
   - All cycles: nx.simple_cycles(G) — complete circular dependency detection
   - Topological sort: nx.topological_sort(G) — optimal analysis ordering

   Files: agents/impact_agent.py, agents/visualizer_agent.py, server/app.py
   New dep: networkx
   Effort: Low — graph data structure already exists, just convert format
   Impact: HIGH — transitive impact alone is a major feature upgrade

--------------------------------------------------------------------------------

2. bandit — Deterministic Python Security Scanning
   Currently: agents/security_agent.py is 100% LLM-based. Files truncated to 3000 chars,
   sent in batches of 4. Findings are non-deterministic, miss issues outside truncation window.

   Recommendation: Run Bandit programmatically as a first pass before the LLM scan:
     from bandit.core import manager as b_manager
     mgr = b_manager.BanditManager(config, "file")
     mgr.discover_files([path])
     mgr.run_tests()
     issues = mgr.get_issue_list()  # CWE-mapped, severity-rated

   - Merge Bandit findings (deterministic, full-file) with LLM findings (semantic, contextual)
   - Skip sending files to LLM if Bandit already found issues (saves tokens)
   - Covers: eval, exec, pickle, yaml.load, hardcoded passwords, weak crypto, SQL injection,
     shell injection — all CWE-tagged

   Files: agents/security_agent.py
   New dep: bandit
   Effort: Low — clean Python API, structured JSON output
   Impact: HIGH — deterministic, zero-cost, full-file coverage

--------------------------------------------------------------------------------

3. radon — Real Cyclomatic Complexity Metrics
   Currently: agents/visualizer_agent.py uses: tokens + imports*10 + functions*5.
   A 500-token file with 5 trivial getters scores the same as one with deeply nested logic.

   Recommendation: Use radon for real cyclomatic complexity per function:
     from radon.complexity import cc_visit
     results = cc_visit(source_code)  # per-function CC scores, grades A-F

   - Feed CC scores into the dependency graph heatmap
   - Include CC grades in the executive report ("3 functions rated F — high maintenance risk")
   - Use CC in impact analysis risk rating

   Files: agents/visualizer_agent.py, agents/impact_agent.py, agents/report_agent.py
   New dep: radon
   Effort: Low — one function call per file, returns structured results
   Impact: MEDIUM — industry-standard metric replaces naive heuristic
   Note: Python-only. Other languages fall back to linear formula.

--------------------------------------------------------------------------------

4. vulture — Better Dead Code Detection
   Currently: agents/dead_code_agent.py uses regex name matching. Doesn't understand
   __all__, attribute access on objects, or dynamic dispatch. Already reduced from 28 to
   4 false positives via decorator detection, but fundamental approach is limited.

   Recommendation: Run vulture as a complement to the existing detector:
     from vulture import Vulture
     v = Vulture()
     v.scan(paths)
     unused = v.get_unused_code()  # includes confidence scores

   - Vulture uses AST analysis (not regex), understands __all__, tracks attribute access
   - Merge results: intersection = high-confidence dead code, union = all candidates
   - Also catches dead imports (current detector only checks functions/classes)

   Files: agents/dead_code_agent.py
   New dep: vulture
   Effort: Low — simple API, runs on file paths
   Impact: MEDIUM — eliminates remaining false positive classes
   Note: Python-only. Existing regex detector still handles other languages.

--------------------------------------------------------------------------------

5. __all__ Extraction in Tree-sitter Parser
   Currently: agents/parser_agent.py extracts functions, classes, imports. Does not extract
   __all__ lists — dead code detection flags re-exported symbols.

   Recommendation: During the tree-sitter walk, detect __all__ = [...] assignment nodes.
   Extract string literals from the list. Store as structure["exports"]. Dead code detector
   skips any symbol in exports.

   Files: agents/parser_agent.py, agents/dead_code_agent.py
   Effort: Low — tree-sitter already parses assignment nodes
   Impact: LOW-MEDIUM — only affects Python libraries using __all__, but those are
           exactly where dead code detection matters most

6. Structured LLM Output (JSON Mode / Tool Use)
   Currently: All LLM interactions rely on prompt engineering for format. Sometimes models
   produce prose instead of the structured format (FILE/SEVERITY/ISSUE/CODE/FIX).

   Recommendation: Use provider-native structured output:
   - Groq/OpenAI/NVIDIA: response_format={"type": "json_schema", "json_schema": {...}}
   - Anthropic: Tool use with a Pydantic-derived schema
   - Gemini: response_schema in GenerateContentConfig

   Define Pydantic models:
     class FileAnalysis(BaseModel):
         purpose: str
         functions: list[FunctionDoc]
         gotchas: list[str]

     class SecurityFinding(BaseModel):
         file: str
         severity: Literal["CRITICAL", "HIGH", "MEDIUM", "LOW"]
         issue: str
         code: str
         fix: str

   Files: config/llm_client.py, agents/analyst_agent.py, agents/security_agent.py
   Effort: Medium — each provider has a different API; need abstraction in llm_client
   Impact: HIGH — eliminates format variance, makes batch merging reliable

7. Multi-Language Import Resolution (JS/TS First)
   Currently: agents/visualizer_agent.py only handles Python from X import Y / import X.
   For JS/TS repos, the resolver doesn't understand import { foo } from './utils'
   or require('./config').

   Recommendation: Add language-specific resolvers:
   - JS/TS: Parse from '...' / require('...'), resolve ./ relative paths,
     handle index.js convention, strip extensions
   - Go: Parse "package/path" import specs, map to directory structure
   - Rust: Parse use crate:: / use super:: declarations

   Files: agents/visualizer_agent.py
   Effort: Medium — each language needs its own resolver, patterns are well-defined
   Impact: HIGH — Arkhe claims 7-language support but dependency graph only works
           correctly for Python

--------------------------------------------------------------------------------

8. Semgrep for Multi-Language Security
   Currently: Bandit (#2) only covers Python. For JS/TS/Go/Java/Ruby repos, security
   scanning is still 100% LLM.

   Recommendation: Run Semgrep with community rules:
     semgrep --config=auto --json target_dir/
   - Free community ruleset covers OWASP Top 10 across 30+ languages
   - AST-based matching (not regex) — far fewer false positives
   - Structured JSON output with CWE tags, severity, fix suggestions
   - Runs via subprocess (Go binary)

   Files: agents/security_agent.py
   New dep: semgrep (system binary or pip)
   Effort: Medium — subprocess call + JSON parsing; rules are pre-built
   Impact: HIGH — multi-language deterministic security scanning at zero LLM cost

--------------------------------------------------------------------------------

9. Call Graph Extraction from Tree-sitter
   Currently: agents/parser_agent.py extracts function/class definitions but not call
   relationships. Dead code, test gap, and impact analysis all operate at file level,
   not function level.

   Recommendation: Extend _walk() to track call_expression nodes:
   - Python: call -> function -> identifier (callee name)
   - JS/TS: call_expression -> function -> identifier
   - Go: call_expression -> function -> identifier

   Track which enclosing function contains each call.
   Store as: structure["calls"] = {"func_a": ["func_b", "func_c"]}

   Enables:
   - Function-level dead code: "function _helper is defined but never called"
   - Function-level impact: "changing parse_file affects parse_modules which calls run"
   - Indirect test coverage: "test calls analyze_parallel -> _analyze_file has indirect coverage"

   Files: agents/parser_agent.py, then dead_code_agent.py, test_gap_agent.py, impact_agent.py
   Effort: Medium — node type mapping per language + scope tracking during walk
   Impact: HIGH — upgrades 3 agents from file-level to function-level granularity

10. Cache Token Counts in SQLite
    Currently: scripts/scan_codebase.py runs tiktoken on every file on every scan.
    SQLite cache stores content hash but token counts are recomputed from scratch.

    Recommendation: Check file_cache for existing token count before running tiktoken.
    The table already has a tokens column — just query it during scan.

    Files: scripts/scan_codebase.py, cache/db.py
    Effort: Low
    Impact: MEDIUM — reduces scan from seconds to milliseconds on incremental runs

11. Parallel Tree-sitter Parsing
    Currently: parser_agent.py:parse_modules() is sequential: [parse_file(f) for f in files]

    Recommendation: Use concurrent.futures.ThreadPoolExecutor — tree-sitter is C-backed
    and releases the GIL.

    Files: agents/parser_agent.py
    Effort: Low — replace list comprehension with executor.map()
    Impact: LOW-MEDIUM — 3-4x speedup on 1000+ file repos

12. Analysis Quality Scoring
    Currently: No way to tell if an LLM analysis result is good or garbage.
    Only check is "not empty."

    Recommendation: Score LLM output by:
    - Does the response mention actual function names from the AST?
    - Is response length proportional to file complexity?
    - Does it contain expected sections (Purpose, Key functions)?
    Low-scoring results get re-analyzed with a different model.

    Files: agents/analyst_agent.py
    Effort: Low — heuristic scoring on existing data
    Impact: MEDIUM — prevents garbage-in-garbage-out for the synthesizer
